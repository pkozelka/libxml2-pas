Header translation rules for libxml2-pas project

Petr Kozelka (C) 2002

This document should help to make all the header translations in the libxml2-pas project constistent and easy to update.

Library structure

  • is used to name
  • The library is designed as a single unit, so that the user does not have to list many names in the "uses" part
  • Each library has a base name (for instance: libxml2, libxslt, iconv) which is also assigned to the unit (for instance: libxml2.pas)
  • Each original header file is translated into one file; its name consists of a prefix (= base name), an underscore, and the original file name, as in this example: libxslt_transform.inc.

Structure of a translated file

The primary goal of translation is, to keep the organization of definitions as similar to the original as possible, with respect to limitations of Pascal compiler.

  • each file begins with a CVS compatability signature line
  • there should be a comment informing about the author of the translation
  • macros are translated as functions; please check carefully whether the macro is really necessary to translate. These functions should be implemented in the main file of the respective library in a section delimited by an explaining comment.

Compatibility signature line

is an information that will help a lot when the original API changes. Using this information, the translation maintainer will always know which version of original was translated; therefore, he can use cvs diff command to find changes, and adopt them into translations.

The syntax is shown in this example (taken from libxml2_tree.inc file):

// CVS-REV:gnome-xml/include/libxml/tree.h:1.96:

where "1.96" is CVS version of the original file and the filename says, which header file is it translated from. Now, the maintainer can look at current CVS version of the tree.h file, see that its current CVS version is (let us say) 1.99, so he makes diff between version 1.96 and 1.99 which gives him a good overview of changes that need to be adopted into translated file.

The compatibility signature line is currently used by the script "checkApiChanges.sh" that retrieves log file, diff file and header file for the new version. This tool helps a lot to the maintainer; look at its initial comments.
